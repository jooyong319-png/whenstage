# 결정 기록

## 왜 gcalen을 접고 새로 시작했나 (2026-07-21)
사용자가 다니는 회사가 gcalen과 거의 동일한 컨셉(게임 정보·쿠폰으로 유저를 모으는
캘린더 서비스, "쌀먹닷컴")을 사내에서 준비 중이었음. gcalen을 회사 동료에게
소개했는데 회사가 비밀유지계약서(겸업금지·기밀유출 조항)를 근거로 gcalen 운영
중단을 요청. 사용자가 수용해 gcalen 서비스는 닫기로 함(저장소는 비공개 전환
예정, 코드는 보존).

다만 `gcalen.com` 도메인과 Vercel 프로젝트는 사용자 소유이므로 재사용하기로
하고, 완전히 다른 주제로 사이트를 새로 만들기로 함.

## 왜 콘서트/음원 발매 캘린더인가
후보 2개 중 선택:
1. 쇼핑/이커머스 세일 캘린더 — 탈락. "쿠폰/딜로 유저를 모은다"는 메커니즘이
   이번에 문제 됐던 것과 똑같아서 회사와 또 비슷한 마찰이 생길 위험이 있음.
2. **콘서트·내한·음원 발매 캘린더 — 채택.** 쿠폰 개념이 아예 없는 순수 일정
   추적이라 회사 사업과 겹치지 않고, K팝 팬덤의 해외 검색 수요가 커서 글로벌
   SEO 확장에도 유리함.

## 왜 쿠폰/게임 허브 기능을 통째로 제거했나
gcalen에는 `data/coupons.json` 기반 쿠폰 카탈로그·게임 허브(`/coupons`,
`/games/[key]`) 기능이 있었는데, 이게 바로 회사와 마찰이 생긴 "정보+쿠폰으로
유저 모으기" 메커니즘 그 자체였음. 새 프로젝트에서는 이 패턴을 완전히
배제하기 위해 관련 코드(`lib/coupons.ts`, `lib/game-hub.ts`,
`components/CouponList.tsx`, `/coupons`, `/games` 라우트)를 전부 삭제.
쿠폰 없이 순수 "일정 추적 + 알림"만 남김.
⚠️ **이 패턴(정보+쿠폰/딜로 유저 모으기)은 앞으로도 절대 다시 넣지 않는다** — 법적
마찰의 직접 원인이었으므로, 기획 담당 프롬프트도 이 방향의 기능은 제안조차 하지 말 것.

콘서트·내한 / 음원 발매(컴백) / 페스티벌 / 팬미팅 — 사용자가 직접 선택.

그대로 재사용(같은 필터 UI·색상 범례 컴포넌트를 그대로 씀).

## 사이트 표시명
~~"콘서트 캘린더"(임시)~~ → **"WhenStage"로 확정(2026-07-22)**. 전체 코드베이스에서
"콘서트 캘린더"/"콘서트캘린더" 문자열을 전수 치환(privacy/terms/about/contact/guide/blog

## data/concerts.*.json 필드명을 안 바꾼 이유
`Game` 인터페이스(`name`, `developer`, `publisher`, `platforms` 등)를 콘서트
도메인에 맞게 전부 리네임하지 않고 그대로 재사용(의미만 재해석:
developer→아티스트/기획사, publisher→주최, platforms→공연장 등). 수십 개
컴포넌트에 걸친 필드명 리네임은 비용 대비 효율이 낮다고 판단, 제품 방향이
검증된 뒤 별도 리팩터링으로 미룸. **아직도 유효한 판단 — 필드명 리네임은
우선순위 낮음, 섣불리 손대지 말 것.**

## ko/en/ja는 번역이 아니라 완전히 독립된 데이터다 (2026-07-21~)
초기엔 gcalen 방식(한국어 원본 + en/ja 선택적 번역)을 검토했으나, 실제로 한국
공연과 미국/일본 공연은 서로 다른 실사 이벤트라 "번역"이라는 개념 자체가
안 맞는다고 판단해 완전 독립 데이터 구조로 확정. `data/concerts.ko/en/ja.json`은
각자 다른 실제 공연을 담고, 리서처 3명이 각자 자기 언어권만 담당(자세한 내용은
`AGENTS.md`). 이후 뉴스(`content/news/`)도 같은 원칙으로 리팩터링, 아티스트
프로필(`data/artist-images.json`/`artist-bios.json`)도 로케일별 독립.

**블로그(모아보기)도 결국 같은 원칙으로 전환(2026-07-22)** — 처음엔 "번역해도 의미가
통하는 콘텐츠라 굳이 3배로 따로 쓸 필요 없다"고 KO 원본 + 선택적 EN/JA 번역 구조로 예외를
뒀었으나, 다른 모든 콘텐츠가 로케일별 독립인데 블로그만 예외인 게 일관성이 떨어진다고
판단해 철회. `content/blog/<slug>.ko.md`/`.en.md`/`.ja.md`로 파일명 규칙을 뉴스와 동일하게
바꾸고, 각 로케일 담당(`prompts/BLOG_RESEARCHER_KO/EN/JA.md`)이 자기 언어권 콘서트 데이터로
독자적인 소재를 골라 쓴다 — 번역이 아니라 완전히 다른 글일 수 있음. 이 결정 당시 블로그
글은 KO 1건뿐이라 마이그레이션 부담 없이 바로 전환.

## 아티스트/공연장 허브 페이지는 콘서트 이미지에서 분리 관리 (2026-07-22)
아티스트 대표 이미지를 처음엔 콘서트 엔트리의 `image_url`에서 재활용했는데,
행사 스냅샷(레드카펫 단체샷 등)이 그대로 대표 이미지가 되는 품질 문제가 있어서
`data/artist-images.json`/`artist-bios.json`으로 완전히 분리. 공연장 페이지
(`/venue`)는 `platforms` 필드를 정규화해 그룹핑하는 방식으로 별도 데이터 파일 없이 구현.

## 콘텐츠 자동화와 기획/개발 자동화는 완전히 분리한다 (2026-07-22)
- **콘텐츠 파이프라인**(리서처 6종 + `prompts/ARTIST_PROFILE.md` + `prompts/BLOG_RESEARCHER_KO/EN/JA.md`): 정해진
  스키마 안에서 데이터/글을 채우는 작업이라 자동 실행 리스크가 낮음. `prompts/PLANNER.md`가 주 1회
  신호(블로그 최신성, 아티스트 프로필 커버리지)를 점검해 필요하면 위 두 개를 대신 실행.
- **기획/개발 파이프라인**(`prompts/PRODUCT_PLANNER.md` + `prompts/PRODUCT_DEVELOPER.md`): 실제 코드·UI·
  기능을 바꾸는 작업이라 리스크가 다름. 콘텐츠 데이터는 절대 건드리지 않고, 이 위키를
  반드시 먼저 읽고 시작한다.
- `prompts/PLANNER.md`와 `prompts/PRODUCT_PLANNER.md`는 이름이 비슷하지만 역할이 다르다 — 헷갈리지 말 것.

## 2차 디자인 오버홀 — 색은 "라이트=모노, 다크=레이저 워시" (2026-07-23)
1차 오버홀(2026-07-22~23)에서 정한 핑크→보라 그라데이션 + Space Grotesk + pill(999px) 라운드
조합을, 오너가 "모던하고 현대적인 느낌"을 원한다고 밝히면서 다시 갈아엎었다. 재검토 과정에서
그 조합 자체가 문제였다는 게 드러남 — 흰 배경 위 보라 계열 그라데이션 히어로, 무난한 지오메트릭
디스플레이 폰트, 어디에나 붙는 pill 라운드는 AI가 별생각 없이 뽑는 "기본값"에 가장 가까운
조합이었다. 그래서 유행을 더 얹는 대신 콘서트라는 소재 자체(무대 조명, 티켓의 물성)에서
재료를 가져오기로 함.

색 후보 4개(리소 포스터=종이+리소블루, 커튼콜=마룬+스칼렛, 레이저 워시=네이비+시안,
모노+의미색=순흑백+카테고리색만)를 시안으로 제안했고, 오너가 **다크 모드=레이저 워시,
라이트 모드=모노**로 조합해서 가자고 결정. 근거: 다크는 공연 당일 저녁에 몰입해서 보는
상황, 라이트는 낮에 빠르게 확인·캡처·공유하는 상황이라 실제 두 사용 맥락과 맞아떨어짐.
라이트 모드는 액센트 색 자체가 없는 대신(잉크=검정이 곧 액센트) "시간에 민감한 정보"
(공연일·D-day·예매 시각)에만 별도의 `--accent-warm`(호박색 계열, 테마 공통)을 씀 — 이건
브랜드 액센트가 아니라 의미색이라 모노 컨셉과 충돌하지 않는다는 판단.

형태도 함께 바꿈: pill(반경 999px)을 어디에나 쓰던 것을 3/6/10/16px "티켓 스텁" 스케일로
좁히고, 블러 그림자 의존을 줄이고 실선 보더·점선 절취선을 1차 구획 수단으로 삼았다.
디스플레이 서체(Space Grotesk)는 제거하고 이미 프로젝트에 있는 Pretendard Variable을
굵기 축(45~920)만으로 본문/디스플레이 겸용으로 쓴다 — 폰트 패밀리가 하나로 줄어 로딩
부담도 줄고, "안전한 지오메트릭 서체"라는 상투성에서도 벗어남. 숫자(날짜·D-day·타임존)는
시스템 모노스페이스(`--font-mono`)로 분리해 전광판/티켓 인쇄 느낌 + 자릿수 정렬을 확보.

**같은 작업 범위에서 기술부채도 함께 정리**(오너가 명시적으로 "같이 정리" 선택) — gcalen
잔재 죽은 CSS/타입(`EventType`의 `game_show` 등), orphan 컴포넌트 7종, 미정의 CSS 변수,
카드 컴포넌트 보더 규칙 불일치를 이번 커밋 범위에 함께 넣음. 자세한 배경은
`wiki/design-audit.md`, 실제 변경 내역은 `PROJECT_STATUS.md` 2026-07-23 항목 참고.
